# WebMux Flutter/Backend Investigation Report

## Problem
When Chat mode is used with OpenCode in tmux, messages can be sent from WebMux Chat to tmux (`inputViaTmux` works), but replies are not consistently received in the correct chat.

Two specific failure modes were reported:

1. After `/new` in OpenCode, chat stops receiving replies.
2. With multiple tmux sessions in the same directory (for example `agent` and `temp`), responses are routed incorrectly (latest/newer session receives replies, older chat does not, or sessions cross).

## Goal
Ensure each WebMux chat (`sessionName + windowIndex`) receives only the replies generated by its own OpenCode process/session, including after `/new`, without cross-session contamination when multiple tmux sessions share the same working directory.

## Environment / Scope
- Frontend location: `flutter/`
- Backend location: `backend-rust/`
- Date of investigation: **March 5, 2026**
- Active tmux sessions included: `agent`, `temp`, others
- Both `agent` and `temp` were running in:
  - `/home/cleber_rodrigues/kiro-bot`

## High-Level Findings

### Confirmed issues found in logs
1. **Single global chat watcher per WebSocket** was replacing previous watcher.
   - Evidence: repeated `Stopping previous chat log watcher` immediately after starting watch for another session.
2. **Opencode PID re-detection by CWD** could hop to another session when multiple OpenCode processes shared same folder.
3. **Session selection mismatch for temp process**:
   - Backend selected stale session id `ses_340a832faffezJhef3BTeRzUHQ`.
   - Actual OpenCode process log showed active session id `ses_340a832f8ffeTl3bkuiVpbycsi`.

### Confirmed non-root-cause
- Cleaning internal `chat_events.db` is **not** the core fix for this routing bug.
- Old rows in OpenCode DB are expected; main issue was incorrect runtime session binding.

## Detailed Evidence (Key Log Points)

### 1) Global watcher replacement (earlier runs)
- Log pattern repeatedly showed:
  - `Starting chat log watch for temp:0`
  - `Stopping previous chat log watcher`
- This proved only one active watcher existed per WebSocket state.

### 2) After per-session watcher fix, still wrong mapping in some cases
- New logs showed both watchers could exist (no global stop), but temp still sometimes did not receive expected updates.

### 3) Hard mismatch between selected session and actual active session
From backend pane logs (`build:1.0`):
- For temp PID `1570486`, backend selected:
  - `Found Opencode session: ses_340a832faffezJhef3BTeRzUHQ`

From OpenCode process log (`~/.local/share/opencode/log/2026-03-05T185203.log`):
- Same process activity used:
  - `sessionID=ses_340a832f8ffeTl3bkuiVpbycsi`
  - `POST /session/ses_340a832f8ffeTl3bkuiVpbycsi/message`

This directly explains why chat was attached to stale thread data for temp.

## What Was Tried (Chronological)

### A) `/new` handling improvements
- Added per-client timestamp signal when `/new` is sent (`InputViaTmux`) so watcher can detect valid session switch window.
- Added guarded session switch logic in OpenCode parser/watcher with tests.

### B) Initial session selection heuristic upgrade
- Replaced simple “latest by directory” fallback with process-start lineage strategy:
  - choose root near process start
  - choose latest node in that root lineage
- Added tests for lineage behavior and switch-reset behavior.

### C) Watcher architecture change (major)
- Replaced single `chat_log_handle` with watcher map keyed by `(session_name, window_index)`.
- `WatchChatLog` now replaces only same-key watcher.
- `/new` signal scoped to same watcher key.

### D) PID re-detection scope fix
- Removed global “find any opencode in cwd” behavior.
- Re-detection now constrained to **pane process tree** (`pane_pid` descendants only).

### E) Active OpenCode session hinting from runtime logs (latest)
- Added detection of active session id from the opencode process open log tail (`sessionID=ses_...`).
- Passes `active_session_id` hint into parser init.
- Parser now has `init_opencode_state_with_hint(...)`:
  - prefer hinted session id if valid in same directory
  - fallback to lineage selection if hint missing/invalid

## Code Areas Changed

### Backend websocket watcher management
- `backend-rust/src/websocket/mod.rs`
  - per `(session_name, window_index)` watcher map
  - key-scoped `/new` registration
  - safe watcher cleanup handling

### OpenCode watch/detection logic
- `backend-rust/src/chat_log/watcher.rs`
  - pane-scoped PID detection/re-detection
  - active OpenCode session hint extraction from process log
  - pass `pane_pid` + `active_session_id` in `AiTool::Opencode`

### OpenCode parser session selection
- `backend-rust/src/chat_log/opencode_parser.rs`
  - `init_opencode_state_with_hint(...)`
  - `select_session_by_id(...)`
  - keep lineage fallback strategy
  - expanded tests

### Shared model
- `backend-rust/src/chat_log/mod.rs`
  - `AiTool::Opencode` now includes:
    - `pane_pid`
    - `active_session_id: Option<String>`

## Database Review

### `chat_events.db`
- Path: `backend-rust/chat_events.db`
- Purpose: persisted chat events by `(session_name, window_index)` for merge/history.
- Conclusion: not the primary source of wrong OpenCode response routing.

### `opencode.db`
- Path: `~/.local/share/opencode/opencode.db`
- Contains many historical session rows in same directory (normal).
- In this case, stale and active session rows coexisted for temp process.
- Core issue was selecting wrong row for live process, not DB corruption.
